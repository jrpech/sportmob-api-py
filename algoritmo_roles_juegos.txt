Explicacion base para armar calendario de juegos

Para armar un calendario de juegos, pueden existir uno o mas torneos registrados, la fuente base para armar roles de juegos, 
parten de lo siguiente:

- El torneo posee una fecha de inicio el cual nos servira para el inicio de la fecha de los partidos


Las espacios donde se podran disputar los juegos estan dados de la siguiente manera

Para fines practicos los torneos inician y finalizan iguales.

El modelo cancha donde se mantienen los registros de canchas tiene relacion a una cuentaId, la tabla figura asi

La tabla se llama asi en la base de datos sportmob_v2.cancha

ID                INT PK
- nombreCancha      VARCHAR(45)
- ubicacionCancha   VARCHAR(100)
- estadoCancha      TINYINT
- ultimaModificacion DATETIME
- idUsuario         INT
- usuarioC          INT
- cuentaId          INT
- fechas            INT

Cada cancha tiene una relacion de fechas que son dias que acepta cada cancha, la tabla se encuentra asi

La tabla se llama asi en la base de datos sportmob_v2.fechas

- id (INT, PK)
- dia (VARCHAR(150)) --> aqui va el dia que puede ser Lunes, Martes, Miercoles, Jueves, Viernes, Sabado, Domingo
- horarios (INT) --> este va null siempre
- categoriaID (INT) --> este valor siempre puede ser cero
- canchaID (INT) --> esta columna es la relacion directa a la cancha


Ahora cada fechas tiene una relacion a una tabla que se llama horarios

La tabla se llama asi en la base de datos sportmob_v2.horarios

- id (INT, PK)
- horario (VARCHAR(150)) --> este valor va en este formato 0001-01-01T00:00:00
- fechaID (INT) --> esta columna es la relacion directa a la fechas


Hasta este punto esta es la forma en la que se pueden obtener las disponibilidades por cuentaId

actualmente yo tengo un metodo en c# con el que puedo obtener los grupos para un torneo

//Metodo que se encarga de buscar a las parejas del torneo separando por grupos si esta aplicado desde el torneo
public List<Grupo> getParejasRRByTorneo(int torneoID)
{
    //Repositorios
    using var equipoR = new EquipoRepository();
    using var torneoR = new TorneoRepository();

    //retorno
    var retorno = new List<Grupo>();

    //Informacion del torneo
    var infoTorneo = torneoR.getById(torneoID);

    //Consultamos la config del torneo para saber si aplica la separacion de grupos
    var dividirGrupos = infoTorneo.dividirRR ?? false;

    //Buscamos a las parejas registrados
    var parejasRegistradas = equipoR.getXTorneo(torneoID);

    if (dividirGrupos == true)
    {
        var noGruposRR = infoTorneo.noGruposRR ?? 2;

        for (int i = 0; i < noGruposRR; i++)
        {
            var equiposGrupo = parejasRegistradas.Where(p => p.grupoRR == (19283891 + i)).ToList();

            var item = new Grupo
            {
                id = 19283891 + i,
                nombre = $"GRUPO {i + 1} de {equiposGrupo.Count} PAREJAS",
                equiposGrupo = equiposGrupo
            };

            retorno.Add(item);
        }
    }
    else
    {
        var equiposTodos = parejasRegistradas.Where(p => p.grupoRR == 19283890).ToList();

        var item = new Grupo();

        item.id = 19283890;
        item.nombre = $"GRUPO GRAL {equiposTodos.Count} PAREJAS";
        item.equiposGrupo = equiposTodos;

        retorno.Add(item);

    }

    return retorno;
}

- equipoID -> equipo.id

Para poder buscar la disponibilidad de cada jugador para el torneo se busca a traves de una tabla

La tabla se llama sportmob_v2.TorneoDisponibilidadJugador;

Descripción:
Almacena los horarios en los que un jugador indica que puede participar dentro de un torneo específico.

Relaciones:
- idTorneo -> torneo.id
- idJugador -> jugador.id

Campos:
- id (INT, PK)
- idTorneo (INT)
- idJugador (INT)
- fecha (DATETIME)
- hora (VARCHAR(50))

Índices:
- idx_torneo (idTorneo)
- idx_jugador (idJugador)
- idx_torneo_jugador (idTorneo, idJugador)

Ahora la estructura en bd de un partido esta de la siguiente forma

La tabla se llama sportmob_v2.partidosjornadas;

Descripción:
Representa un partido programado entre dos equipos dentro de un torneo.

Relaciones:
- idJornada -> jornada.id
- idEquipo1 -> equipo.id
- idEquipo2 -> equipo.id
- lugar -> cancha.id ocupa el id de la cancha
- grupoID -> grupo.id -> esta es la que va a ser la referencia a getParejasRRByTorneo que son los grupos del torneo

Campos principales:

Identificación
- id (INT, PK)

Participantes
- idEquipo1 (INT)
- idEquipo2 (INT)
- grupoID (INT) -> esta es la que va a ser la referencia a getParejasRRByTorneo que son los grupos del torneo

Programación
- fechaHoraPartido (DATETIME)
- hora (VARCHAR)
- lugar (INT)
- etiquetaCancha (VARCHAR)

Resultado
- marcadorEquipo1 (INT)
- marcadorEquipo2 (INT)
- estadoPartido (VARCHAR)

Control
- tipoPartido (VARCHAR)
- partidoSugerido (TINYINT)
- modificadoManual (TINYINT)
- currentStatePadel (VARCHAR)


Restricciones de Calendarización
- Los partidos únicamente pueden ocurrir entre equipos del mismo grupo.
- Nunca pueden existir enfrentamientos entre grupos distintos.
- La disponibilidad de una pareja se obtiene mediante la intersección de sus jugadores.
- Un partido solamente puede programarse cuando existe una intersección entre:
     Disponibilidad Equipo A
     Disponibilidad Equipo B
     Slot disponible
- Un slot solamente puede contener un partido.
- Una pareja NO puede jugar dos partidos en el mismo horario.
- Una pareja SÍ puede jugar múltiples partidos el mismo día siempre que los horarios sean distintos.
- Los partidos pueden adelantarse si existe disponibilidad y hay slots libres.
- No es obligatorio respetar jornadas rígidas.
- El objetivo principal es completar la mayor cantidad posible de partidos
